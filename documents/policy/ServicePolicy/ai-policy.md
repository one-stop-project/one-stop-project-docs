### AI 리뷰 요약

- 리뷰 5개 이상 누적 시 자동 트리거 (비동기) 또는 관리자 강제 갱신
- 평균 별점 + 최신순 최대 50건 리뷰 기반 AI 요약 제공
- 카테고리별 프롬프트 분기
    - 의류 → 착용감 / 사이즈 / 소재 중심
    - 전자제품 → 성능 / 배터리 / 호환성 중심
    - 식품 → 맛 / 식감 / 신선도 중심
    - 일반 → 범용 요약
- 요약 결과 DB 저장, 조회 시 DB 직접 조회 (Redis 캐시 없음)
- AI 장애 시 DB에 저장된 기존 요약 반환 (Fallback)
- 리뷰 5개 미만: INSUFFICIENT 상태 반환
- 요약 미생성 상태: PENDING 상태 반환
- 요약 미생성 상태: PENDING 상태 반환
- 증분 업데이트: 리뷰 작성 이벤트마다 비동기 실행
    - 새 리뷰 5개 미만 누적 시 스킵 (다음 이벤트 대기)
    - 새 리뷰 20건 초과 시 전체 재요약으로 폴백
    - 리뷰 수정/삭제 시 전체 재요약
    - 낙관적 락 충돌 시 1회 재시도

### AI 추천 상품

- 연관 상품 추천 (인기순), 현재 구현은 DB 기반
- 동일 카테고리 내 후보 최대 30개 조회 → 재고 필터링 → 판매수 기준 TOP 10 반환

- 품절 / 비활성 상품 제외

### AI 장애 격리 공통

- Resilience4j Circuit Breaker + Fallback 적용
- 카테고리/후보 상품 없을 시 인기상품 반환
- "타임아웃 시 고정 안내 메시지 반환”
- Rate Limit 초과 시 Circuit Breaker 동작
- 프롬프트 외부화 (코드 하드코딩 금지, 재배포 없이 수정 가능)
- 토큰 로깅: requestId / promptTokens / latencyMs / model / status /completionTokns

## AI 더미데이터

### 1. 범위

- 1차 생성 대상: 상품만 (= 리뷰·주문은 1차 제외, 추후 별도 논의)
- 구조 요건: 확장 가능하게 설계 (= 공용 생성 엔진 + 도메인별 writer 분리 → 리뷰·주문 writer를 나중에 추가)

### 2. 데이터 출처

- 상품 원본: 네이버 쇼핑 "검색 API" (= 네이버 개발자센터 오픈 검색 API. 스마트스토어/커머스 판매자 API 아님)
- API가 주는 필드: `title` · `brand` · `maker` · `category1~4` · `lprice` · `image` · `mallName` · `productType`
- 상품명: 상품명: LLM이 변형 정보(용량·색상 등) 제거한 깔끔한 대표명 생성 (실패 시 원본 정제 fallback). ※멱등·중복 식별 키는 원본 raw 필드 기반 — LLM 생성물은 식별키에 넣지 않음(재실행 안정성)

### 3. 네이버 API 한도 & 상품 생성 가능량

| 항목 | 값 |
| --- | --- |
| 일일 호출 한도 | **25,000회 / 일** (무료 등록 키 기준) |
| 호출당 최대 건수 | `display` **최대 100건** (기본 10) |
| 페이징 한계 | `start` **최대 1,000** → 한 검색어로 최대 1,000건 |
- 생성 가능량: 소 카테고리 60개를 검색어로 → 이론상 최대 약 60,000건(중복 제거 전), 호출 ~600회 = 일일 한도의 약 2.4%
- → 네이버 API는 우리 규모에서 상한이 아님. 실질 병목은 아래 "비용"의 LLM
- ※ 키 발급 후 개발자센터 콘솔에서 본인 키 실제 한도 1회 확인 해야함

### 4. 수량

- 1차 목표: 약 1,000개
- 추가 생성: LLM 비용 부담 없으면 1차 이후 더 늘릴 수 있음 (네이버 한도는 여유) — 상한 `[확인 후 업데이트 예정]`
- 세부 분배·생성 주기(1회성/반복): 작성자가 확정

### 5. 비용 정책 (=  LLM 비용)

- 네이버 검색 API: 무료. 상품 시드 비용 부담 없음
- LLM(설명 그라운딩, 향후 리뷰 생성)
    - 1차 상품 description: 그라운딩(짧은 사실 변환)이라 토큰 소모 적음 → 1차 부담 낮음
    - 리뷰·주문 등 확장 더미데이터: LLM 호출량 큼 → `[확인 후 결정]`
- 비용 통제 원칙: 무료 티어(예: Gemini 무료 한도) 내 우선 시도 → 초과분은 추후 논의 후 결정 (※ 무료 한도는 사용 시점 콘솔 확인)

### 6. 카테고리 (확정: 대 6 / 중 18 / 소 60 = 84 노드)

- 식품
    - 신선식품 → 과일 / 채소 / 정육·계란
    - 가공·간편식 → 라면·면류 / 통조림·즉석밥 / 냉동식품
    - 음료·기호식품 → 커피·차 / 생수·음료 / 과자·스낵 / 건강식품
- 전자제품
    - 모바일·PC → 스마트폰 / 노트북 / 태블릿
    - 생활가전 → TV / 냉장고 / 세탁기 / 청소기
    - 음향·주변기기 → 이어폰·헤드폰 / 키보드·마우스 / 모니터
- 의류
    - 여성의류 → 원피스 / 상의(블라우스·티) / 아우터
    - 남성의류 → 셔츠 / 팬츠 / 점퍼·코트
    - 잡화·언더웨어 → 신발 / 가방 / 모자 / 속옷·양말
- 도서
    - 문학 → 소설 / 시·에세이 / 자기계발
    - 실용·전문 → 경제·경영 / IT·컴퓨터 / 자격증·수험서
    - 교양·기타 → 어린이·유아 / 만화 / 잡지 / 외국도서
- 반려동물 용품
    - 사료·간식 → 강아지 사료 / 고양이 사료 / 간식·영양제
    - 용품 → 장난감 / 하우스·방석 / 이동장·캐리어 / 식기·급수기
    - 위생·미용 → 배변패드 / 미용·목욕 / 의류·액세서리
- 화장품
    - 스킨케어 → 스킨·토너 / 로션·에센스 / 크림 / 선케어
    - 메이크업 → 베이스·파운데이션 / 립 / 아이
    - 클렌징·바디·헤어 → 클렌징 / 바디케어 / 헤어케어

### 7. 상품설명(description) — 사실 기반 그라운딩 (= 환각 차단)

- 원칙: LLM에 창작 권한 없음. 주어진 실제 사실만 문장화 (= 입력에 없는 스펙·수치는 생성 금지)
- 1차 기본(전 상품, 항상 동작): 네이버 필드(`brand`·`maker`·`category1~4`·`lprice`·`title`)를 그라운딩해 설명 생성
    - 1차 = LLM 그라운딩(변형 그룹핑 LLM 호출에서 이름·옵션과 함께 생성. 사실만, 창작 금지).
      LLM 실패 시 네이버 필드 템플릿 fallback(NOT NULL 보장).
      공식 스펙 기반 풍부한 설명 = 2차 보강
    - description은 NOT NULL + FULLTEXT 대상 (그대로)
- 2차 보강 `[확인 후 결정]`: 정형 스펙이 존재하는 카테고리(예: 전자제품)에 한해, 브랜드 공식 스펙을 출처로 상세 보강
    - 선결 검토: ① 공식페이지 커버리지 ② 사이트별 파싱 ③ 크롤링 ToS·차단 ④ 비용·호출량
- 상세 이미지 / 풀스펙 = 2차(옵션 다양화와 함께)
- ※ description은 NOT NULL이자 FULLTEXT 검색 대상이라 비울 수 없음

### 8. 중복 제거 (= 동일 상품이 여러 건 들어가는 것 방지)

- 1차(규칙 기반): `productType`(= 가격비교 카탈로그 "대표" 항목만 채택) + `brand`+`maker`+정규화한 `title` 조합
    - (정규화 = 공백 제거·소문자 통일 후 비교)
- 판매 방식이 다르면 별개 상품으로 취급 (= 동일 모델이라도 묶지 않음)
    - 예: 휴대폰 "자급제" vs "통신사향(통신사 지정 개통)"은 서로 다른 상품
- 2차(LLM): 단순 표기 차이(띄어쓰기·오타 수준)만 같은 상품으로 판별
- 멱등성: base 그룹키(규칙 결정적 = brand+maker+변형토큰 제거 title) ↔ Product / listing
  키(NAVER productId 우선) ↔ ProductItem. 재실행 시 기존이면 INSERT 막고 가격만 갱신, 새 변형은 ProductItem
  추가. (productId는 그룹 식별엔 부적합 → 변형 식별용으로만)
- ※ `productId`는 몰마다 달라 단독 기준으로 쓰지 않음

### 9. 이미지

- 저장: 네이버 이미지 URL을 ImageStorage로 다운로드 저장 (1단계 로컬, 추후 S3 어댑터 교체. 외부 URL 직접 참조 안 함). 다운로드 실패·누락 시 기본 이미지 → 교체완료
- 누락 대비: 기본 이미지 1종("상품 이미지 준비중입니다…") → 이미지 없으면 대체
- 1차는 썸네일 1장만 (= 상세 다중 이미지는 2차)

### 10. 옵션 (ProductItem) — 2단계 처리

- 변형 그룹핑으로 실제 옵션 생성: 네이버가 별개 listing으로 주는 같은 모델의 변형(용량/색상)을 1 Product + 여러 ProductItem(옵션 조합)으로 묶음.
- 옵션 축 이름·값·옵션별 가격은 LLM이 실제 변형 listing에서 추출(창작 아님 = 환각 0).
- 규칙이 그룹 경계·키를 결정적으로 잡고 LLM은 옵션·이름·설명만 채움.
- 변형이 하나뿐이면 무옵션 ProductItem 1개.

### 11. 판매자 (소유 계정)

- 더미 상품 전용 SELLER 계정 1개 (= 예: "AI상점", 승인(APPROVED) 상태)
    - (= 상품은 판매자 소유라 seller FK 필수. 관리자(ADMIN)는 상품 소유 개념이 없어 사용 불가)
    - 판매자 권한은 "자기 상품만"이라 별도 권한 축소 불필요

### 12. 생성·승인 방식

- 더미 상품은 생성 즉시 승인(APPROVED) 상태 (= 승인 안 되면 검색에 노출 안 됨)
- 승인: 시드 도구가 도메인 메서드 `approve()`를 직접 호출 (= 관리자 승인 API 미경유 → 관리자 이력 오염 방지)
- DB 투입: 기존 상품 등록 도메인 로직 재사용 (= 직접 SQL INSERT 아님)
- 실행 형태: 스케줄러(@Scheduled 매일 1회 + dummy.scheduler.enabled 플래그 + naver.api.client-id 조건부). 플래그 off면 빈 자체가 안 뜸.

### 13. 생성 전 선결 정합성 (= 시드 실패 방지)

- 더미 판매자 계정이 APPROVED 상태여야 함
- 상품마다 ProductItem 1건 이상(가격·재고 NOT NULL) 생성
- 카테고리 레코드가 먼저 존재해야 하고, 상품-카테고리 매핑은 1~3개 (= @Size min 1, max 3)
