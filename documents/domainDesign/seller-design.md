# Seller 도메인 설계문서

## 1. 도메인 목적

Seller 도메인은 판매자의 입점, 승인, 판매자 전용 상품/리뷰/대시보드 조회를 담당한다.

구매자 중심의 User 도메인과 분리하여, 판매자에게 필요한 비즈니스 정보와 권한 검증을 독립적으로 관리한다.

## 2. 주요 책임

| 책임 | 설명 |
|---|---|
| 판매자 등록 | 사업자 정보 기반 판매자 신청 |
| 판매자 승인 상태 관리 | 승인 전/승인 후 접근 제어 |
| 판매자 정보 조회 | 내 판매자 정보 조회 |
| 판매자 상품 관리 | 판매자 본인 상품 기준 조회/관리 |
| 판매자 리뷰 조회 | 본인 상품에 달린 리뷰만 조회 |
| 판매자 대시보드 | 매출, 주문, 리뷰 등 요약 정보 제공 |
| 소유권 검증 | 다른 판매자의 리소스 접근 차단 |

## 3. 주요 흐름

### 3.1 판매자 등록

```text
인증 사용자 요청
↓
기존 판매자 등록 여부 확인
↓
사업자/정산 정보 검증
↓
Seller 생성
↓
승인 대기 상태 저장
```

### 3.2 승인된 판매자 API 접근

```text
요청 사용자 확인
↓
Seller 조회
↓
승인 상태 확인
↓
판매자 전용 API 처리
```

### 3.3 판매자 상품 리뷰 조회

```text
판매자 인증
↓
상품 조회
↓
상품의 sellerId와 요청 sellerId 비교
↓
일치하면 리뷰 조회
↓
불일치하면 접근 거부
```

### Seller Dashboard View

대시보드는 별도 엔티티라기보다 주문/결제/상품/리뷰 데이터를 판매자 기준으로 집계한 읽기 모델에 가깝다.

## 5. 기술 선택 근거

### 판매자 승인 상태 검증

판매자 등록과 판매자 API 이용 가능 상태는 다르다. 입점 신청 직후에는 판매자 엔티티가 존재하지만, 승인 전에는 상품 등록이나 대시보드 접근을 제한해야 한다.

따라서 판매자 API 진입 시 `getApprovedSeller(userId)` 같은 공통 조회 메서드로 승인 상태를 검증한다.

### PageResponse 사용

Spring Data `Page<T>`를 API 응답으로 직접 노출하면 `pageable`, `sort`, `unpaged` 등 프레임워크 내부 구조가 응답에 섞인다.

이를 방지하기 위해 `PageResponse<T>`로 감싸 클라이언트에 필요한 필드만 제공한다.

### 소유권 검증

판매자 API는 IDOR 취약점이 발생하기 쉽다. 예를 들어 판매자가 URL의 productId만 바꿔 다른 판매자의 상품 리뷰를 조회할 수 있으면 안 된다.

따라서 상품/리뷰/주문 조회 시 항상 요청 판매자와 리소스 소유자를 비교한다.

## 6. 정합성/보안 고려사항

- 판매자 전용 API는 SELLER 권한 또는 승인된 Seller 여부를 확인한다.
- 승인되지 않은 판매자는 대시보드/상품 관리에 접근할 수 없다.
- 상품, 리뷰, 주문 조회 시 sellerId 소유권을 검증한다.
- Page 객체를 직접 노출하지 않는다.
- 정산 계좌 등 민감 정보는 응답 범위를 제한한다.
- 관리자 승인/거절 이력은 별도 감사 대상으로 관리할 수 있다.

## 7. 예외 처리

| 상황 | 처리 |
|---|---|
| 판매자 정보 없음 | SELLER_NOT_FOUND |
| 승인되지 않은 판매자 | SELLER_NOT_APPROVED |
| 다른 판매자 리소스 접근 | ACCESS_DENIED |
| 중복 판매자 등록 | DUPLICATED_SELLER |
| 잘못된 사업자 정보 | INVALID_BUSINESS_INFO |

## 8. 한계 및 후속 개선

| 한계 | 후속 개선 |
|---|---|
| 승인 프로세스 단순 | 관리자 승인/반려 사유 및 이력 고도화 |
| 대시보드 실시간 집계 비용 가능성 | 통계 테이블 또는 배치 집계 도입 |
| 소유권 위반 감사로그 제한 | SELLER_IDOR_ATTEMPT 보안감사로그 추가 |
| 정산 정보 보안 강화 필요 | 계좌 정보 암호화 및 마스킹 응답 적용 |
