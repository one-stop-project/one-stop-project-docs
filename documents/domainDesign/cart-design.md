# Cart 도메인 설계문서

## 1. 도메인 목적

Cart 도메인은 회원 및 비회원 사용자의 장바구니 상품 관리와 로그인 시 장바구니 병합을 담당한다.

장바구니는 구매 전환율과 직접 연결되므로 사용자가 로그인하지 않은 상태에서도 데이터를 유지할 수 있어야 한다.

## 2. 주요 책임

| 책임 | 설명 |
|---|---|
| 회원 장바구니 | DB 기반 사용자 장바구니 관리 |
| 비회원 장바구니 | Redis 기반 임시 장바구니 관리 |
| 상품 추가/수정/삭제 | 장바구니 아이템 관리 |
| 로그인 병합 | 비회원 장바구니를 회원 장바구니로 병합 |
| TTL 관리 | 비회원 장바구니 자동 만료 |

## 3. 주요 흐름

### 3.1 비회원 장바구니 추가

```text
비회원 상품 추가 요청
↓
guest_cart_id 쿠키 확인
↓
없으면 UUID 발급
↓
Redis Hash에 상품/수량 저장
↓
TTL 갱신
```

### 3.2 회원 장바구니 추가

```text
인증 사용자 요청
↓
회원 Cart 조회 또는 생성
↓
기존 상품 여부 확인
↓
수량 증가 또는 신규 CartItem 생성
↓
DB 저장
```

### 3.3 로그인 시 장바구니 병합

```text
로그인 성공
↓
guest_cart_id 쿠키 확인
↓
Redis 비회원 장바구니 조회
↓
회원 장바구니 조회
↓
동일 상품은 수량 병합
↓
비회원 장바구니 삭제
↓
guest_cart_id 쿠키 제거
```

## 4. 데이터 모델

### Cart

| 필드 | 설명 |
|---|---|
| id | 장바구니 ID |
| userId | 회원 사용자 ID |
| createdAt/updatedAt | 생성/수정 시각 |

### CartItem

| 필드 | 설명 |
|---|---|
| id | 장바구니 아이템 ID |
| cartId | 장바구니 ID |
| productId | 상품 ID |
| quantity | 수량 |

### Guest Cart Redis

```text
cart:guest:{guestCartId}
  productId -> quantity
```

## 5. 기술 선택 근거

### Redis Hash for Guest Cart

초기에는 Cookie에 장바구니 데이터를 직접 저장할 수 있지만, Cookie는 용량 제한이 있고 요청마다 전송되어 네트워크 비용이 증가한다.

따라서 Cookie에는 `guest_cart_id`만 저장하고 실제 장바구니 데이터는 Redis Hash에 저장한다.

장점:

- Cookie Payload 감소
- 데이터 위변조 위험 감소
- TTL 기반 자동 만료 가능
- 로그인 병합 처리 용이

### DB for Member Cart

회원 장바구니는 장기간 유지되어야 하며, 주문 전환과 연결되므로 RDB에 저장한다.

## 6. 정합성 고려사항

- 동일 상품 추가 시 중복 row 대신 수량을 증가시킨다.
- 로그인 병합 시 동일 상품 수량 합산 정책을 명확히 한다.
- 상품 판매 상태가 변경된 경우 장바구니 조회 시 검증한다.
- 비회원 장바구니 TTL을 갱신하여 장기간 방치 데이터를 자동 정리한다.

## 7. 예외 처리

| 상황 | 처리 |
|---|---|
| 존재하지 않는 상품 | PRODUCT_NOT_FOUND |
| 판매 불가 상품 | PRODUCT_NOT_AVAILABLE |
| 수량 제한 초과 | INVALID_QUANTITY |
| 비회원 장바구니 없음 | 빈 장바구니 처리 |
| 병합 중 상품 비활성화 | 해당 상품 제외 또는 알림 |

## 8. 한계 및 후속 개선

| 한계 | 후속 개선 |
|---|---|
| Redis 장애 시 비회원 장바구니 사용 불가 | graceful fallback 또는 안내 메시지 |
| 병합 정책 단순 | 품절/가격변경 사용자 알림 추가 |
| 장바구니 가격 스냅샷 제한 | 조회 시 현재 가격과 담은 시점 가격 비교 |
| 대량 장바구니 정리 제한 | 만료/비활성 장바구니 정리 배치 추가 |
